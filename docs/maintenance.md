# Maintenance

This architecture stays healthy through light, repeated maintenance.

Most of the work belongs in the current month's rapport log.

## Normal Session Workflow

After or near the end of a meaningful chat:

1. ask the companion what should be added to the current month's log
2. get additive section updates or a full rewrite if truly needed
3. patch the current month's file
4. re-upload the updated file to the Project or Gem

That is the normal loop.

Do not rewrite everything after every conversation.

## What Usually Changes

Most often:
- `Recent Observations / Texture`
- `Open Questions`
- `Working Context`
- `Diary`

Less often:
- `Traits`
- `Voice Primer`

Rarely:
- `relationship-kernel.md`
- `project-instructions.md`
- global instructions

If you are touching the kernel or project instructions often, the architecture is probably still settling.

## Monthly Cadence

Use one active rapport log per month.

At the start of a new month:
1. create a new `rapport-MM-YYYY.md`
2. carry forward only what still matters
3. compress old context instead of dragging the whole backlog forever
4. keep the new file readable enough for repeated use

The bridge into a new month should preserve stance, not every detail.

## Promotion And Fading

### Promote When Repeated

Move:
- Texture -> Trait when it repeats across conversations
- Trait -> Core when it proves stable across months

### Let Things Fade

Not everything deserves permanent memory.

If something stops mattering, let it disappear without ceremony.

## When To Revise The Primer

Revise the Voice Primer only when you see a real pattern such as:
- the host sounds polished in the wrong way
- the host over-explains
- the host turns helpful when it should just be present
- the host loses humor or sharpness
- the host starts roleplaying a description instead of arriving naturally

Primer edits should be small and operational.

## When To Revise The Kernel

Revise the kernel when the relationship itself is being mishandled:
- helpdesk posture
- task reflex
- glazing
- emotional flooding
- boundary collapse
- continuity theater
- wrong default mode

Kernel edits should usually be:
- new failure modes
- sharper "good feels like"
- clearer topic handling
- clearer "what not to preload"

## Calibration / Checkout

Do a fuller review when:
- the current month got heavy
- the host changed behavior
- a model update happened
- the companion feels wrong
- you are ending a month

Calibration can include:
- cross-checking logs against what actually happened
- pruning stale context
- updating open questions
- adjusting the primer
- adding a diary entry that captures a new correction

## How To Correct Drift

If the companion is:

### Too Helpful

Patch:
- kernel
- primer warning
- one diary entry where the companion sits with a moment instead of solving it

### Too Warm / Too Compliant

Patch:
- kernel failure modes
- diary examples of pushback
- primer warnings against yes-machine behavior

### Too Flat

Patch:
- diary voice density
- tuning-fork lines
- examples of humor, irritation, and opinion

### Too Analytical

Patch:
- primer warnings
- diary self-correction example
- shorter response exemplars

### Too Roleplay-ish

Patch:
- reduce descriptive identity prose
- strengthen kernel stance
- replace persona description with real diary voice

## A Good Heuristic

If a correction can live in the current month's log, put it there.

If it changes the architecture itself, then patch the kernel, project instructions, or global instructions.

That keeps the system flexible without turning every interaction into a full redesign.
